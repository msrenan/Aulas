7. Descreva técnicas usadas pelo sistema operacional para **interagir** com os **controladores** de dispositivos. Resposta: O SO interage com controladores de dispositivos através de interrupções. Quando um dispositivo externo, como um dispositivo de entrada/saída, a placa de rede ou o controlador de disco por exemplo, é acionado ou o SO está pedindo alguma informação, ou o controlador de disco está devolvendo uma informação. Se o SO está pedindo uma informação, o SO já está no controle da CPU, e manda um pedido de informação padronizado para o controlador necessário. Agora, se um programa está em execução e um dispositivo externo terminou de produzir / procurar a informação, ele dispara uma interrupção via hardware, o que faz o processo em execução sair de cena para dar lugar ao SO, que executará uma das suas rotinas de tratamento de interrupção para decidir o que fazer com a informação entregue pelo controlador, e após isso, chama o escalonador para que o fluxo normal de execução dos processos continue. Outra maneira do SO de interagir com controladores é ficar lendo repetidamente um registrador de status do controlador, para verificar se ele já terminou. É ineficiente comparado às interrupções. Por fim, para grandes volumes de dados o SO utiliza o DMA (**Direct Memory Access**), configurando um controlador especial, o chip DMA com o endereço de origem, destino e tamanho. O DMA faz uma cópia dos dados diretamente para a RAM sem a CPU precisar intervir. Ao fim da cópia, o DMA gera apenas uma interrupção avisando que os dados estão prontos na RAM.

10. Como deve ser uma rotina de **tratamento de interrupção** no sistema operacional? Que ações devem ser executadas? Resposta: Uma rotina de tratamento de interrupção deve primeiro identificar o motivo pelo qual foi acionada, e depois fazer as mediações necessárias para resolver o problema ou entregar a informação solicitada. Essa mediações podem envolver pedido de dados para dispositivos externos, chamadas de sistemas, entre outros. Dito isso, mais tecnicamente, em uma interrupção, a primeira ação é salvar o contexto do processo corrente para que nada seja perdido durante a troca de processos em execução (já que o SO vai entrar em cena), depois deve ser enviado um sinal ao controlador de interrupção (um ACK), avisando que a interrupção foi recebida. Depois disso, a rotina de tratamento é executada o mais rápida o possível. Por fim, o contexto do processo em execução é restaurado e o SO sai de cena.
11. O que são **chamadas de sistema**? Como são implementadas e de que maneira um programa de usuário tem acesso aos serviços do sistema operacional? Uma chamada de sistema é uma função que do SO que interage diretamente com o kernel para realizar alguma ação. Os programas que precisam de input do usuário, mapeamento de memória, criação de threads e outras diversas ações que envolvem, diretamente, um contato maior com o kernel, não podem fazer essa interação por conta própria, e por tanto, pedem para que o SO faça a interação e apenas propague o resultado. As syscalls são implementadas através de wrapper functions disponibilizadas pela libc, em C, fazendo com que o usuário apenas tenha que chamar um função com um parâmetro contendo a informação, e por baixo dos panos a libc ajeita os registradores e invoca chama `syscall()`.

20. Diferencie as chamadas **fork()** e **vfork()**. Resposta: A chamada `fork()` usada para criar um processo filho, faz com que todo o task_struct as áreas de endereçamento de um processo pai sejam **copiadas** para o processo filho, um processo custoso (hoje em dia, uma otimização foi feita, o *CoW* - Copy on Write: as áreas de endereçamento do processo pai só são copiadas para o processo filho se um dos dois processos for fazer uma modificação nela, caso contrario, as áreas de memória do filho apontam para as do pai.) Já a chamada `vfork()` é uma chamada otimizada, usada quando um processo filho usa o `exec()` logo após sua criação (e sabendo que a chamada `exec()` descarta as áreas de memória atuais de um processo para criar novas áreas para o programa que será executado, não fazia sentido todo o processo de cópia do processo pai para o filho). Assim, um processo filho criado usando `vfork()` tem no lugar de suas áreas de memória um ponteiro para as áreas de memória do pai, que para evitar conflitos de IPC, fica paralisado até que o filho deixe de utilizar as suas áreas de memória, ou seja: até que o filho execute o `exec()` ou termine sua execução com `exit()`.
